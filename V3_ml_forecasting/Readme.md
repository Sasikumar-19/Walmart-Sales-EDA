# Walmart Sales Forecasting — EDA v3
 
**45 stores · 4,320 training rows · 1,935 test rows · 20 engineered features**
 
Builds directly on EDA v2. The feature-engineered dataset (lag features,
rolling averages, holiday flags) from v2 becomes the training input for three
ML forecasting models here. v3 answers the question v2 couldn't:
*can we predict next week's sales — and which factors actually drive them?*
 
---
 
## Dependency on v2
 
v3 cannot run standalone. It reads `walmart_features.csv` generated by
`EDA_v2_Walmart.ipynb`. Run v2 first.
 
```
EDA_v2_Walmart.ipynb    → produces walmart_features.csv
EDA_v3_ML_Forecasting.ipynb  → reads walmart_features.csv
```

## Model Results (Test Set — 2012 data, unseen during training)
 
| Model | MAE | RMSE | MAPE | R² |
|---|---|---|---|---|
| **Gradient Boosting** | **$36,247** | **$53,956** | **3.57%** | **0.9899** |
| Random Forest | $45,651 | $67,226 | 4.41% | 0.9843 |
| Ridge Regression | $65,605 | $91,421 | 7.80% | 0.9710 |
 
**Winner: Gradient Boosting** — R² of 0.99 means the model explains 99% of
the variance in weekly sales on unseen 2012 data. A MAPE of 3.57% means the
average prediction is off by $37K on a store averaging ~$1M/week — well within
a useful margin for operational planning.

## What This Demonstrates
 
| Concept | Where it appears |
|---|---|
| **Data leakage prevention** | Time-based train/test split (train: 2010–2011, test: 2012) — no random split on a time series |
| **Feature engineering value** | Lag + rolling features drive 95%+ of model performance |
| **Model pipeline** | `StandardScaler + Ridge` in a single sklearn Pipeline object |
| **Ensemble methods** | Random Forest (bagging) vs Gradient Boosting (sequential boosting) |
| **Feature importance comparison** | RF vs GB side-by-side — both agree on the top predictor |
| **Rolling forecast** | Each week's prediction feeds the next week's lag feature |
| **Honest baseline** | Ridge Regression included deliberately to show what a linear model can and can't do |
 
---
 
## Train / Test Split Rationale
 
```
Train: Feb 2010 – Dec 2011 (4,320 rows, ~2 full years)
Test:  Jan 2012 – Oct 2012 (1,935 rows, ~10 months)
```
 
A random split would contaminate the test set with future information —
for example, a December 2011 row randomly assigned to training tells the
model what December sales look like before it's asked to predict November 2012.
Time-based split prevents this. The model only ever sees data from before
its prediction horizon.
